# MPI Checkpoint Restart — Problem, Diagnosis, and Fix

## Why Checkpoint Restart

At fidelity level 7 (L7, 2¹⁴ = 16 384 max cells) each simulation runs for
~80 non-dimensional mixing cycles.  On OSCAR, a single L7 job takes 0.64 h
on 16 MPI ranks.  At L8 the estimate grows to 24 h, which already saturates
the 24-h partition walltime limit.  Checkpoint restart splits any job into a
chain of segments, each within walltime, and hands state forward via a
`dump`/`restore` pair.  Without it, L8 sweeps are not feasible.

---

## What Was Going Wrong

Three independent bugs blocked a working checkpoint chain.  They are listed in
discovery order; all were found through falsifiable computational experiments
tracked in `experiments/hypothesis_ledger.json`.

### Bug 1 — Multigrid blow-up on restart (binary level)

**Symptom:** A 2-rank MPI restart from a valid checkpoint produced
`oxy = Inf` within a few time steps after `t_mix`, even though a 1-rank restart
from the same file produced finite values.

**Root cause (two coupled mechanisms):**

1. **Stale pool-scalar ghost cells in `h_residual`.**
   `foreach_cell() res[] = 0` in `mg_cycle` initialises the residual scalar on
   owned cells, but it silently skips MPI ghost cells at coarse levels that
   have no fine-level descendants on the local rank.  Those cells retained
   garbage values (~−1.42 × 10²⁰) from a previous `mg_solve` call (the
   pressure Poisson solve).  When `h_residual` later restricted the residual
   upward, the garbage became the right-hand side `b[]` fed to `h_relax`.

   **Fix (`bc1b92e`):** call `restriction({res})` immediately after the
   `foreach_cell` zero-init.  This propagates the all-zero leaf state to every
   coarse ghost cell via `halo_restriction` before any real residual is
   computed.

2. **`struct HDiffusion q` declared without initialiser.**
   `h_relax` declares `struct HDiffusion q;` without `= {0}`.  The member
   `q.embed_flux` therefore holds stack garbage.  The guard
   ```c
   if (!q.embed_flux && aa.boundary[embed] != symmetry)
   ```
   was silently skipped whenever the garbage was non-NULL.  For the oxygen
   scalar, which uses the default `symmetry` embed boundary condition,
   `if (p->embed_flux)` then fired at coarse multigrid levels where the
   face-fraction `fa ≈ 0`.  The resulting division
   `mua / (fa + SEPS) × oxy` overflowed to `8.87 × 10¹⁸`, and the
   multigrid diverged at `t_mix`.

   **Fix (`6f15883`):** add `q.embed_flux = NULL` immediately after the
   struct declaration.

**Supporting production fixes** (same commit range, retained permanently):

| Fix | Rationale |
|---|---|
| `fedisableexcept` wrap in `h_relax` | Prevents SIGFPE from masking the real blow-up location during diagnosis; harmless in production |
| `restriction({res})` in `h_residual` | Zero-initialises coarse ghost cells (see above) |
| Harmonic mean for diffusion coefficient `D` at coarse levels | Avoids spurious large `D` at faces adjacent to embedded walls |
| Stencil pre-extraction in `h_relax` | Closes a qcc static-stencil blind-spot that caused MPI ghost cells to receive zero instead of the actual field value under certain branch predicates |
| `d`-floor guard and `isfinite(n)` guard in `h_relax` | Belt-and-suspenders: clamps pathological values before they leave the relaxation sweep |

**Validation:** 2-rank restart from a fidelity-5 checkpoint passes `t_mix`
with `oxy ≈ 0.229` at `t = 103.9`.  See ledger entry H10–H11 and commit
`6f15883`.

### Bug 2 — Chain self-submission died after segment 1

**Symptom:** After each seg-0 run completed, the SLURM template tried to
submit seg-1.  Seg-1 ran fine.  But seg-1 never submitted seg-2; the chain
stopped there.

**Root cause:** `slurm_mpi_template.sh` derived the canonical run directory of
the *next* segment as:

```bash
NEXT_CANON="$(dirname "$CANON_RUN")/$NEXT_RUN"
```

`CANON_RUN` was read from the field `_canonical_run_dir` in `params.json`.
That field is written by `simulate.py` only for the *first* segment (seg-0);
subsequent params files are generated by the sweep infrastructure without it.
So for any seg-1 or later run, `CANON_RUN=""`, `dirname ""` evaluates to `.`,
`NEXT_CANON` became `./next_run_id`, the existence check failed, and the
`sbatch` call was never reached.

**Fix (`db4e12f`):** derive `RUNS_ROOT` (the parent of all canonical run dirs)
from `_experiment_dir` in `params.json`, which *is* propagated through every
segment.  See also Bug 4 — the initial version of this fix had an off-by-one
dirname that was corrected in a follow-up commit.

### Bug 4 — RUNS_ROOT resolved to `experiments/runs/` instead of `runs/`

**Symptom:** Chains stopped at seg-3.  `squeue` showed the job was submitted
and completed, but no seg-3 result appeared and no seg-4 was queued.  The
SLURM log showed no error — the template silently skipped the `sbatch` call.

**Root cause:** The Bug 2 fix computed `RUNS_ROOT` as:

```python
os.path.join(os.path.dirname(exp), 'runs')
```

`_experiment_dir` is a path like
`…/rocking-bioreactor-2d/experiments/sweep_fb_theta_l7_mpi_ckpt/`.
`os.path.dirname(exp)` therefore resolves to `…/experiments/`, not the
project root.  So `RUNS_ROOT` became `…/experiments/runs/`, which does not
exist.  The guard

```bash
if [ -f "$NEXT_PARAMS_CANON" ]; then
```

silently evaluated to false and the `sbatch` was never reached.

**Fix (`0a6a66f`):** add one extra `dirname` call so the path climbs to the
project root (parent of `experiments/`):

```bash
RUNS_ROOT=$(python3 -c "
import json, sys, os
p = json.load(open(sys.argv[1]))
exp = p.get('_experiment_dir', '')
if exp:
    print(os.path.join(os.path.dirname(os.path.dirname(exp)), 'runs'))
" "$PARAMS" 2>/dev/null)
[ -z "$RUNS_ROOT" ] && [ -n "$CANON_RUN" ] && RUNS_ROOT="$(dirname "$CANON_RUN")"
NEXT_CANON="$RUNS_ROOT/$NEXT_RUN"
```

`dirname(dirname(_experiment_dir))` = `…/rocking-bioreactor-2d/` →
`RUNS_ROOT = …/rocking-bioreactor-2d/runs/` (correct).

### Bug 3 — Postprocessing path resolved to `/oscar/scratch`

**Symptom:** Jobs completed their simulation (checkpoint written, MPI
collective exits cleanly) but SLURM reported `ExitCode 2` and no
`results.json` appeared.  The error log showed:

```
can't open file '/oscar/scratch/scripts/postprocess.py': [Errno 13] Permission denied
```

**Root cause:** The postprocessing block in the template has two branches.
The first branch (correct path) executes when `CANON_RUN` is non-empty:

```bash
if [ -n "$CANON_RUN" ] && [ "$CANON_RUN" != "$SCRATCH_RUN" ]; then
    PROJECT_ROOT="$(dirname "$(dirname "$CANON_RUN")")"
    uv run python "$PROJECT_ROOT/scripts/postprocess.py" "$CANON_RUN"
```

The second branch (fallback) executes otherwise:

```bash
else
    PROJECT_ROOT="$(cd "$SCRATCH_RUN/../../.." && pwd)"
    uv run python "$PROJECT_ROOT/scripts/postprocess.py" "$SCRATCH_RUN"
```

Chain-submitted jobs (all segments after seg-0) arrive with `CANON_RUN=""`
because Bug 2 was patched before it was known that the postprocess branch also
depended on `CANON_RUN`.  `SCRATCH_RUN` is
`/oscar/scratch/eaguerov/mpi_runs/<run_id>`, so `../../..` resolves to
`/oscar/scratch` — an unrelated scratch root with no `scripts/` directory and
no read permission for the project venv.

**Fix (`8236ea7`):** derive `CANON_RUN` from `_experiment_dir` before the
postprocessing block, ensuring the first branch is always taken for chain jobs:

```bash
if [ -z "$CANON_RUN" ]; then
    _EXP_DIR=$(python3 -c "import json,sys; p=json.load(open(sys.argv[1]));
                           print(p.get('_experiment_dir',''))" "$PARAMS")
    if [ -n "$_EXP_DIR" ]; then
        _RUN_ID=$(basename "$SCRATCH_RUN")
        CANON_RUN="$(dirname "$(dirname "$_EXP_DIR")")/runs/$_RUN_ID"
    fi
fi
```

The fallback `else` branch is now dead code for any run submitted from a
recognised sweep experiment.

## Validation

After all three fixes, a full L7 sweep was run under MPI checkpoint restart
(binary: `BioReactor-mpi-stripped`, 16 ranks, fidelity 7, 80 mixing cycles
per chain split into 10 × 8-cycle segments) and compared against the
pre-existing single-job baselines:

| Sweep | Conditions | Max |Δ kLa₂₅| / baseline | Mean |Δ kLa₂₅| / baseline |
|---|---|---|---|
| Fill sweep (5 fill × 10 ω) | 50 | 0.08 % | 0.02 % |
| Theta sweep (6 θ × 10 ω) | 60 | 0.00 % | 0.00 % |
| **Combined** | **110** | **0.08 %** | **0.01 %** |

Deviations are consistent with floating-point non-determinism across MPI rank
layouts between the fresh and restarted runs.  No condition exceeded 0.1 %.

Figure: `experiments/figures/checkpoint_validation.pdf` — identity scatter and
residual strip across all 110 conditions.

## Walltime Savings

Measured on the L7 theta sweep (fidelity 7, 16 MPI ranks, chain of 10 segments,
each 8 mixing cycles):

| Run type | Walltime per 8-cycle unit |
|---|---|
| Fresh seg-0 (cold start) | 0.64 h |
| Restart seg-1+ (warm start from checkpoint) | 0.34 h |
| **Saving per restart segment** | **47 %** |

The saving comes from skipping the adaptive refinement spin-up phase that
occupies the first few mixing cycles of every fresh run.  At L8 (estimated
24 h total), a 10-segment chain would reclaim roughly 4 h per 8-cycle
segment — the difference between fitting inside the 24-h partition limit and
exceeding it.

## Bug 5 — Process Never Terminates After the Final Checkpoint Write (short runs)

**Discovered:** 2026-07-08/09, while auditing why 3 of 4 running L10 tau-sweep
seg0 jobs and all 4 submitted L9 runs showed `checkpoint.dump` written hours
earlier with zero further file output since, yet all 16 MPI ranks still at
~99.5% CPU. This is a different bug from Bugs 1–4 above (those were about
restart *correctness*; this one is about fresh-run *termination*), found in a
separate debugging session after the original restart fix above had already
been validated and merged.

**Symptom:** After `dump_checkpoint` writes `checkpoint.dump` at
`t = t_dump_checkpoint` (the period-aligned point at or just after `t_end`),
the binary keeps running indefinitely — full CPU on every rank, zero further
disk writes — until SLURM's walltime limit kills the job. No `results.json`
is ever produced, because the job script's postprocess step only runs after
the binary exits on its own.

**Root cause:** `event acceleration (i++)` and `event oxygen (t=t_mix; i++)`
are *unconditional* Basilisk events — no `t <= t_end` bound. Basilisk's
`run()` loop (`grid/events.h`) only stops once every event with an active
condition or single-shot init has become "finished"; once the four
`t <= t_end`-bounded events (`logstats`, `normcal`, `pressure_diagnostics`,
`movies_output`) expire past `t_dump_checkpoint`, these two unconditional
events are all that's left, and they keep the loop alive forever with no
further observable output.

**Why L7 sweeps never surfaced this distinctly:** L7 production runs have
`t_end ~ 95` (many rocking periods), so a fresh L7 job's *total* wall time
already spans hours — the "extra" post-checkpoint tail, if it existed, was
indistinguishable from normal long-running work within the walltime budget.
L9/L10 tau-only sweeps (hydrodynamics-only, `t_end ~ 1–20`) made the bug
obvious: the useful work finishes in minutes, but the job then silently
burns the *rest* of its multi-day walltime allocation doing nothing.

**Falsification chain** (full detail in `experiments/hypothesis_ledger.json`,
thread `l9-l10-post-checkpoint-hang`) — each of the following was tested and
ruled out before landing on the actual cause:

| Hypothesis | Result |
|---|---|
| `geometry.n=8` (unusual bag shape) triggers it | Falsified — identical hang with `n=2` |
| Oxygen injection never firing (`n_mix_cycles=500`) triggers it | Falsified — identical hang with `n_mix_cycles=1` |
| MPI oversubscription (2 ranks / 1 core) is an artifact | Falsified — identical hang with 4 genuinely dedicated ranks |
| Fidelity (grid size) is the differentiator | Falsified — identical hang at fidelity 6 *and* 7 |
| **No termination condition once `t<=t_end` events expire** | **Confirmed** via temporary per-rank/per-step instrumentation showing `iter`/`t` climbing indefinitely past checkpoint with zero disk writes |

**Fix (commit `90f9b4e`):** have `dump_checkpoint`'s event action `return 1`
immediately after `dump()`. Returning a nonzero value from an event action is
the documented Basilisk idiom for stopping `run()`'s event loop cleanly and
immediately (`grid/events.h`: `event_do()` treats a nonzero return as
`event_stop`) — `main()`'s `fclose()` calls after `run()` still execute
normally, so this is a clean exit, not an abrupt `exit()`/`abort()`.

**Validation:**
- Fidelity 6, `t_end=1.0` (the shortest tested): exit code 0 in 39 s
  (previously hung 100+ s with zero progress at the same point).
- Fidelity 6, `t_end=4.56` (matches the originally-stalled production run
  `61d53bc5` exactly): clean exit with Basilisk's standard closing summary
  printed; `postprocess.py` produced finite `tau_98_qss=0.00517`,
  `tau_100_max=0.0316`, `vor_mean=0.918` (`kLa_25=nan` as expected —
  hydrodynamics-only config, oxygen never injected by design).
- Restart-segment path (`t_checkpoint>0`, continuing from that same
  checkpoint for `t_end=1.5` more): also exited cleanly with its own closing
  summary.
- **Not yet validated** at full production scale (fidelity 9/10, 16
  dedicated ranks) — the fix is only proven at fidelity 6/7 with 2–4 ranks.
  The fixed binary has **not** been deployed to
  `/oscar/scratch/eaguerov/BioReactor-mpi-video` yet (a backup of the
  pre-fix binary was made at `BioReactor-mpi-video.pre-checkpoint-fix-backup`
  in the same directory, but the swap itself was intentionally held for
  explicit confirmation before touching the binary real sweep jobs depend
  on).
